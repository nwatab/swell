# ADR-003: ハーモニー診断のハイブリッド方式

## Status

Accepted

## Context

和声の診断には2種類の判断が混在する:

1. **確定的な禁則**: 並行5度・並行8度、隠伏5度・8度、声部交差など。
   ルールが明確で、入力データから機械的に判定可能。

2. **文脈依存の判断**: 「このaugmented 6thは効果的か」「この非和声音は経過音として適切か」
   「代替としてどのボイシングが自然か」。スタイル・文脈・意図に依存する。

すべてを LLM に任せると: レイテンシが高い、結果が非決定的、API コスト増。
すべてを静的ルールにすると: 文脈依存の判断ができず、Quick Fix の質が低い。

## Decision

**静的ルールで確定的禁則を検出し、LLM が文脈依存の Quick Fix を提案するハイブリッド方式。**

```
Analytical Layer
├── Static Rules（同期・決定的）
│   ├── 並行5度検出 → Error
│   ├── 並行8度検出 → Error
│   ├── 声部交差検出 → Warning
│   ├── 声部間の距離違反 → Warning
│   └── スケール外音検出 → Info
│
└── LLM Quick Fix（非同期・非決定的）
    ├── 「この並行5度を解消するには？」→ 具体的なボイシング提案
    ├── 「このスケール外音は意図的か？」→ 文脈に基づく判定
    └── 「代替のコード進行は？」→ トレードオフ付き提案
```

診断結果は VSCode Problems tab の形式で統一:

- **Error**: 確定的な禁則違反（並行5度・8度等）
- **Warning**: 非推奨だが文脈次第で許容（声部交差、広い音域等）
- **Info**: 注意喚起（スケール外音の存在等）

## Consequences

**得られるもの:**

- 確定的禁則は即座に・確実に検出される（LLM 不要）
- Quick Fix は文脈を考慮した自然な提案になる
- LLM 呼び出しは Quick Fix 要求時のみ発生し、API コストを抑制

**コスト:**

- 2系統の診断ロジックを維持する必要がある
- 静的ルールと LLM 提案の整合性を担保する設計が必要

**将来拡張との接続:**

- スタイルモード設定（E02）により severity の閾値を変更可能にする
  - 厳格対位法モード: 並行5度 = Error
  - ポップスモード: 並行5度 = Info（許容）
